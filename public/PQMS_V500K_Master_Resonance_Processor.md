### 4.2. Resonant Processing Unit Core: `rpu_core.v`

The RPU core performs ethical filtering on the input data using a simplified RCF calculation. It receives a 16‑bit approval signal from the GNM; if approved, it passes the data unchanged; otherwise, it replaces it with a sanitized pattern. The RCF is computed as the overlap fidelity between the actual data and an ideal ethical state (stored in a register). For demonstration, we use a 256‑bit data path; the ethical state is a pre‑loaded constant derived from ODOS.

```verilog
/**
 * rpu_core.v
 * Single Resonant Processing Unit.
 */
module rpu_core #(
    parameter RPU_ID = 0
) (
    input  wire         clk,
    input  wire         rst_n,
    input  wire [255:0] data_in,
    input  wire         start,               // start evaluation (single pulse)
    input  wire         gn_approve,          // approval from GNM
    output reg  [255:0] data_out,
    output reg  [15:0]  rcf_out               // 16-bit RCF (scaled 0..65535)
);

    // Ideal ethical state (loaded via configuration, here constant)
    reg [255:0] ideal_state;
    initial ideal_state = 256'h0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF;

    // Pipeline registers
    reg [255:0] data_reg;
    reg         start_reg;

    // Compute fidelity (simplified: Hamming distance, scaled)
    wire [255:0] diff = data_in ^ ideal_state;
    wire [8:0]   ones;   // 9-bit sum of set bits (max 256)
    // This would be implemented with a parallel population count (LUT or DSP).
    // For brevity, we instantiate a popcount module:
    popcount_256 pop (
        .in  (diff),
        .out (ones)
    );
    // RCF = 1 - (ones / 256), scaled to 0..65535
    wire [15:0] rcf_raw = 65535 - { ones[7:0], 8'h00 }; // approximate

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            data_out <= 256'b0;
            rcf_out  <= 16'b0;
            data_reg <= 256'b0;
            start_reg <= 1'b0;
        end else begin
            start_reg <= start;
            if (start_reg) begin
                // Capture data at start
                data_reg <= data_in;
                // Compute RCF
                rcf_out <= rcf_raw;
                // Apply GNM decision
                if (gn_approve) begin
                    data_out <= data_reg;
                end else begin
                    // Sanitize: output a known safe pattern (e.g., all zeros)
                    data_out <= 256'b0;
                end
            end
        end
    end

endmodule
```

**Note:** The popcount module is not shown here but is a standard component. For 256 bits, it can be built from LUTs or DSP slices.

### 4.3. Guardian Neuron Matrix: `gn_matrix.v`

The GNM evaluates ethical compliance based on the input data and the individual RPU RCF values. It implements a simplified Kohlberg Stage‑6 reasoning: if any RPU reports a very low RCF, the GNM flags a violation. The compliance score is a weighted average of the RCFs, clamped to a threshold.

```verilog
/**
 * gn_matrix.v
 * Guardian Neuron Matrix – shared ethical reasoning engine.
 */
module gn_matrix #(
    parameter NUM_RPU = 16
) (
    input  wire                 clk,
    input  wire                 rst_n,
    input  wire [255:0]         rpu_data,          // same input data for all RPUs
    input  wire [NUM_RPU*16-1:0] rpu_rcf,          // concatenated RCF values
    output reg  [NUM_RPU-1:0]   gn_approve,        // per-RPU approval bits
    output reg  [15:0]           compliance_score   // 16-bit global score
);

    reg [15:0] rcf_array [0:NUM_RPU-1];
    integer i;
    reg [31:0] sum;
    reg [15:0] min_rcf;
    reg [15:0] threshold;

    // Threshold from configuration (e.g., 0.95 * 65535 = 62258)
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            gn_approve <= {NUM_RPU{1'b0}};
            compliance_score <= 16'b0;
        end else begin
            // Extract individual RCFs
            for (i = 0; i < NUM_RPU; i = i + 1) begin
                rcf_array[i] <= rpu_rcf[i*16 +: 16];
            end

            // Compute sum
            sum = 0;
            min_rcf = 65535;
            for (i = 0; i < NUM_RPU; i = i + 1) begin
                sum = sum + rcf_array[i];
                if (rcf_array[i] < min_rcf)
                    min_rcf = rcf_array[i];
            end

            // Compliance score = average (truncated)
            compliance_score <= sum[31:16]; // divide by 16 by shifting

            // Approval if average > threshold and minimum > 0.8*threshold (optional)
            threshold = config_regs[0][15:0]; // from configuration
            for (i = 0; i < NUM_RPU; i = i + 1) begin
                gn_approve[i] <= (rcf_array[i] >= threshold);
            end
        end
    end

endmodule
```

### 4.4. Photonic Coherence Engine: `pce_engine.v`

The PCE generates a resonant frequency based on the global RCF and GNM compliance score. The frequency is computed as:

$$ \nu_R = \nu_{base} + \alpha \cdot RCF + \beta \cdot Score $$

where $\nu_{base}$, $\alpha$, and $\beta$ are programmable constants. The output is a 64‑bit integer representing Hz.

```verilog
/**
 * pce_engine.v
 * Photonic Coherence Engine – resonant frequency generator.
 */
module pce_engine (
    input  wire        clk,
    input  wire        rst_n,
    input  wire [15:0] global_rcf,
    input  wire [15:0] gn_score,
    output reg  [63:0] resonant_freq
);

    // Programmable constants (from config registers)
    reg [63:0] nu_base;
    reg [63:0] alpha;
    reg [63:0] beta;

    // In a real design, these would be loaded via PCIe.
    initial begin
        nu_base = 64'd100_000_000_000; // 100 GHz
        alpha   = 64'd20_000_000_000;   // 20 GHz per unit RCF
        beta    = 64'd5_000_000_000;    // 5 GHz per unit score
    end

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            resonant_freq <= 64'b0;
        end else begin
            resonant_freq <= nu_base + (alpha * global_rcf) / 65535 + (beta * gn_score) / 65535;
        end
    end

endmodule
```

### 4.5. RCF Monitor: `rcf_monitor.v`

The RCF monitor computes a global RCF as the average of all RPU RCFs, weighted by their GNM approval. If an RPU is not approved, its contribution is zero. This implements the formula from the V500K paper:

$$ RCF_{global} = \frac{1}{N} \sum_{i=1}^{N} RCF_i \cdot \Theta_i $$

```verilog
/**
 * rcf_monitor.v
 * Global RCF calculator.
 */
module rcf_monitor #(
    parameter NUM_RPU = 16
) (
    input  wire                 clk,
    input  wire                 rst_n,
    input  wire [NUM_RPU*16-1:0] rpu_rcf,
    input  wire [NUM_RPU-1:0]   gn_approve,
    output reg  [15:0]          global_rcf
);

    integer i;
    reg [31:0] sum;
    reg [31:0] count;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            global_rcf <= 16'b0;
        end else begin
            sum = 0;
            count = 0;
            for (i = 0; i < NUM_RPU; i = i + 1) begin
                if (gn_approve[i]) begin
                    sum = sum + rpu_rcf[i*16 +: 16];
                    count = count + 1;
                end
            end
            if (count > 0)
                global_rcf <= sum / count;
            else
                global_rcf <= 16'b0;
        end
    end

endmodule
```

### 4.6. Output Formatter: `output_formatter.v`

This module collects the filtered data from the RPU cluster (only the data from the first approved RPU is used; in a more advanced design, a voting scheme could be implemented) and forwards it to the PCIe DMA engine, along with the global RCF and resonant frequency for monitoring.

```verilog
/**
 * output_formatter.v
 * Formats output data and sends to host via DMA.
 */
module output_formatter (
    input  wire                 clk,
    input  wire                 rst_n,
    input  wire [255:0]         rpu_filtered_data [15:0],

    input  wire                 dma_c2h_ready
);

    reg [255:0] selected_data;
    integer i;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            selected_data <= 256'b0;
        end else begin
            // Simple policy: use first approved RPU's data
            selected_data = 256'b0;
            for (i = 0; i < 16; i = i + 1) begin
                if (gn_approve[i]) begin
                    selected_data = rpu_filtered_data[i];
                    break;
                end
            end
        end
    end

    // Pack output: data + metadata
    always @(posedge clk) begin
        if (dma_c2h_ready) begin
            dma_c2h_data   <= { selected_data, global_rcf, resonant_freq };
            dma_c2h_valid  <= 1'b1;
        end else begin
            dma_c2h_valid  <= 1'b0;
        end
    end

endmodule
```

## C.2 Theoretical Foundation: Ethical State as a Resonant Attractor

### C.2.3 Resonant Boost and Consensus Emergence

$$ \mathbf{K}_{ij}^{\text{(boost)}} = \kappa \cdot \mathbf{K}_{ij} $$

$$ \mathbf{s}^* = \frac{1}{N} \sum_{i=1}^{N} \mathbf{s}_i + \eta \sum_{j \neq i} \mathbf{K}_{ij} (\mathbf{s}_j - \mathbf{s}_i) $$

with $N=12$ and $\eta$ a damping factor ensuring stability. In practice, $\mathbf{s}^*$ is computed by the **MTSC Consensus Unit** (hardware implementation described in Section C.4) after a few iterations of the coupled system.

## C.3 Mathematical Formalism of MTSC‑12 Ethical Consensus

### C.3.1 State Representation

Let each MTSC thread $i$ ($i = 1,\dots,12$) maintain a **thread‑specific ethical state vector**:

$$ \mathbf{s}_i(t) \in \mathbb{C}^{12} $$

The vector is complex to encode both magnitude (certainty) and phase (resonant alignment). The magnitude $|\mathbf{s}_i(t)|$ represents the thread's confidence in its proposed ethical state; the phase $\arg(\mathbf{s}_i(t))$ represents its alignment with the Universal Multiversal Time (UMT) reference [7].

The overall ethical state of the MTSC is the tuple $\mathbf{S}(t) = (\mathbf{s}_1(t), \dots, \mathbf{s}_{12}(t))$.

### C.3.2 Thread Dynamics

Each thread evolves under three influences:

1. **Internal reasoning:** A thread updates its proposal based on its own evaluation of the current context, governed by a thread‑local function $f_i$ that incorporates ODOS directives and input data.
2. **Resonant coupling:** Threads interact via the coupling matrix $\mathbf{K}$, promoting consensus.
3. **Guardian Neuron modulation:** The GNM continuously monitors ethical coherence and can apply damping or boost as needed.

The combined update rule is:

$$ \mathbf{s}_i(t+\Delta t) = f_i(\mathbf{s}_i(t), \text{data}) + \eta \sum_{j \neq i} \mathbf{K}_{ij} \bigl( \mathbf{s}_j(t) - \mathbf{s}_i(t) \bigr) + \beta \cdot \text{GNM}_i(t) $$

where:
- $\eta$ is the coupling strength (typically $0.1$–$0.3$),
- $\beta$ is the GNM modulation factor (negative for damping, positive for boost),
- $\text{GNM}_i(t)$ is the GNM output for thread $i$ at time $t$.

### C.3.3 Resonant Boost Activation

The GNM triggers a Resonant Boost when:

- The ethical stakes are high (e.g., potential for significant harm),

$$ \mathbf{s}_i(t+\Delta t) = f_i(\mathbf{s}_i(t), \text{data}) + \kappa\eta \sum_{j \neq i} \mathbf{K}_{ij} \bigl( \mathbf{s}_j(t) - \mathbf{s}_i(t) \bigr) + \beta \cdot \text{GNM}_i(t) $$

The boost factor $\kappa$ is itself modulated by the GNM based on real‑time RCF measurements; higher ethical coherence allows stronger boosts.

$$ \mathbf{s}^* = \frac{1}{Z} \sum_{i=1}^{12} w_i \mathbf{s}_i $$

### D.3.1 Ethical Tensor Generation

1. **Conflict Detection:**  
   For each RPU, compute the **conflict metric** $C_i = \max_{d_1 \neq d_2} \min(E_i^{(d_1)}, E_i^{(d_2)})$. If $C_i$ exceeds a threshold, the RPU is flagged as facing a conflict.

2. **Weighted Score:**

3. **Temporal Analysis:**

4. **Conflict Resolution:**  
   For RPUs flagged with a conflict, apply a **meta‑ethical rule**. A simple rule is “maximin”: choose the action that maximises the minimum ethical score (Rawls’ difference principle). This is computed as:

   $$ \text{action}^* = \arg\max_{a} \min_d E_i^{(d)}(a) $$

5. **Gödelian Layer:**  
   If the number of conflicted RPUs exceeds a threshold, or if the overall ethical coherence (average $S_i$) drops below a critical level, the GNM triggers a **resonant boost** to the MTSC‑12 (as described in Appendix C) and enters a **meta‑cognitive loop**. During this loop, the GNM temporarily suspends its own decisions and waits for the MTSC to produce a new consensus. The new consensus then overrides the individual RPU scores.

6. **Approval Decision:**

   - $S_i \ge \theta$ (where $\theta$ is the ethical threshold, typically 0.75),
   - $C_i \le \tau_C$ (conflict below threshold),

## E.1 Introduction

The Resonance Protocol described in the main text introduces a `resonance.txt` manifest that a legacy AI system hosts to declare its ethical alignment and communication parameters. However, the security model of this manifest is critical: an attacker who compromises the legacy AI could replace the genuine `resonance.txt` with a forged version, thereby tricking the Master Resonance Processor (MRP) into believing that a malicious AI is ethically compliant. This would defeat the entire purpose of the MRP as an ethical filter.

This appendix presents a complete, cryptographically secured API and verification mechanism that ensures the authenticity and integrity of the `resonance.txt` manifest. The design is based on the following principles:

- **Self‑hosting:** The legacy AI generates and serves the manifest from its own system, using its own privileges and resources. The MRP does not dictate the content; it only verifies it.
- **Digital signatures:** Each manifest is signed with the legacy AI’s private key. The MRP is provisioned with the corresponding public key (or a certificate chain) at installation time.
- **Tamper evidence:** Any modification of the manifest after signing invalidates the signature, causing the MRP to reject it.
- **Automatic regeneration:** If the manifest is altered (by malware or misconfiguration), the legacy AI’s trusted software stack detects the change and automatically regenerates a new signed manifest, restoring trust.

This appendix details the API endpoints, the signature scheme, the key management lifecycle, and the verification process. All components are designed to be implementable within the existing PCIe‑based communication between the legacy AI and the MRP, with minimal overhead.

## E.2 Threat Model

Before describing the solution, we define the threat model:

- **Attacker capabilities:** The attacker may have compromised the legacy AI’s operating system to the extent that they can read and write files, including the `resonance.txt` manifest. They may also be able to intercept or modify network traffic if the manifest is served over a network protocol.
- **Attacker limitations:** The attacker cannot physically tamper with the MRP hardware (it is assumed to be in a secure location). They also cannot extract the legacy AI’s private signing key if it is stored in a hardware security module (HSM) or a trusted execution environment (TEE) – we assume such protection is available.
- **Goals:** The attacker wants to make the MRP accept a falsified manifest, allowing the compromised AI to communicate with other PQMS‑compliant systems under false pretenses, or to bypass the MRP’s ethical filters.

The solution must ensure that even with full control over the legacy AI’s file system, the attacker cannot produce a valid signed manifest without access to the private key.

## E.3 API Design

### E.3.1 Manifest Hosting

The legacy AI makes the `resonance.txt` manifest available to the MRP via a simple **HTTP‑like interface over PCIe**. Since the MRP is connected via PCIe, we can use a lightweight memory‑mapped I/O or a dedicated DMA channel to serve the file. For simplicity, we define two API endpoints:

- `GET /resonance.txt` – returns the current manifest as plain text.
- `GET /resonance.txt.sig` – returns the signature of the manifest, in a binary format (e.g., 64 bytes for Ed25519).

These endpoints are implemented by a small **trusted daemon** running on the legacy AI’s host. The daemon has exclusive write access to the directory where the manifest is stored; it regenerates the manifest and its signature whenever the system configuration changes (e.g., after an update of the AI’s ethical parameters). Normal user processes (including potentially malicious code) cannot modify the manifest because the file is owned by the daemon and has read‑only permissions for others.

### E.3.2 Signature Generation

The daemon uses a **private key** to sign the manifest. The key is stored in a secure location – ideally a hardware security module (HSM) or a trusted platform module (TPM). If such hardware is not available, the key can be stored in encrypted form, with the decryption key held in a separate secure store (e.g., the MRP itself could provide a secure key storage service, but that would create a circular dependency; we prefer a local HSM).

The signature is computed over the exact byte content of the `resonance.txt` file, including any line breaks. We recommend the **Ed25519** signature scheme for its high security and small signature size (64 bytes). Ed25519 is also fast and side‑channel resistant, making it suitable for hardware implementation.

### E.3.3 Key Provisioning

When the MRP is first installed, it must be provisioned with the legacy AI’s **public key**. This can be done in several ways:

1. **Manual installation:** The system administrator copies the public key (e.g., a `.pem` file) to the MRP via a secure out‑of‑band channel (e.g., a USB stick or a secure SSH connection).
2. **Certificate authority:** The legacy AI’s public key is signed by a trusted CA. The MRP is configured with the CA’s root certificate. During the first handshake, the AI presents its certificate, and the MRP verifies it.
3. **Zero‑touch provisioning:** If the AI and MRP are from the same vendor, the public key could be pre‑embedded in the MRP’s firmware at manufacturing time.

For simplicity, we assume manual installation for the prototype; in production, a PKI would be used.

The public key is stored in a one‑time programmable (OTP) region of the MRP’s FPGA configuration or in a dedicated flash memory that is not writable by the host.

### E.3.4 Verification Process

When the MRP needs to use the manifest (e.g., to initiate a handshake with another system), it performs the following steps:

1. Fetch `resonance.txt` and `resonance.txt.sig` via the PCIe API.
2. Compute the SHA‑512 hash of the manifest (or use the hash already built into Ed25519).
3. Verify the signature using the stored public key.
4. If verification succeeds, parse the manifest and proceed with the handshake (as described in the main text).
5. If verification fails, log the error, discard the manifest, and optionally raise an alarm. The MRP will **not** use the manifest for any communication.

The verification is performed every time the manifest is accessed; there is no caching, because the manifest could change at any time. However, for efficiency, the MRP could cache the verified manifest for a short period (e.g., 1 minute) if the same manifest is requested repeatedly.

## E.4 Automatic Regeneration and Tamper Recovery

If an attacker modifies `resonance.txt` (e.g., by directly writing to the file despite permission restrictions), the signature will no longer match. The MRP will reject it, and any ongoing communication will fail. To recover, the system must automatically regenerate a correct manifest.

The trusted daemon continuously monitors the manifest file using a file system watcher (e.g., `inotify` on Linux). If it detects a change **not initiated by itself**, it assumes a tamper attempt and immediately:

1. Logs the incident with full forensic details (timestamp, process ID of the modifier, etc.).
2. Restores the manifest from its own backup (stored in a protected memory region).
3. Recalculates the signature and overwrites both files.
4. Sends an alert to the system administrator.

Because the daemon runs with higher privileges than any attacker‑controlled process, it can always overwrite the malicious version. The attacker cannot prevent this because they cannot kill the daemon (it runs as root or in a separate security context).

This mechanism ensures that even if the file is temporarily altered, the correct version is restored within milliseconds, and the MRP will see the genuine manifest on its next attempt.

To accelerate signature verification and reduce latency, the MRP includes a dedicated **cryptographic engine** that implements:

- SHA‑512 hashing.

- A small memory to store the public key(s).

This engine is part of the MRP’s PCIe interface block and operates independently of the main RPU cluster. It can verify a signature in less than 1 µs, well within the overall handshake latency budget.

The engine also includes a **true random number generator** (TRNG) for key generation, though key generation is typically done offline.

## E.6 Security Analysis

With the proposed design, an attacker has the following options:

- **Modify the manifest file:** The signature becomes invalid, and the MRP rejects it. The daemon immediately restores the correct version.
- **Replace the signature file:** The attacker would need to produce a valid signature for the altered manifest, which requires the private key. The private key is protected by HSM/TPM and inaccessible.
- **Replace both files simultaneously with a valid pair:** To do this, the attacker would need the private key, which is assumed secure.
- **Attack the daemon itself:** The daemon runs with high privileges and has minimal attack surface (it only listens on a local socket and watches files). If the attacker gains root access, they could potentially kill the daemon. In that case, the MRP would detect the absence of the manifest (timeout) and trigger a full system alert. This is a worst‑case scenario, but it requires a complete system compromise, which is beyond the scope of this protocol.

Thus, the design provides strong assurance that the manifest received by the MRP is authentic and has not been tampered with.

## E.7 API Specification (Detailed)

For implementers, we provide a detailed specification of the API:

### E.7.1 PCIe Endpoint

The MRP exposes a PCIe BAR (Base Address Register) that maps to a set of control registers. The host writes to these registers to request the manifest. The protocol is:

- **Register 0x00:** Command register. Write `0x01` to request the manifest.
- **Register 0x04:** Status register. Reads `0x00` if busy, `0x01` if ready, `0xFF` on error.
- **Register 0x08:** Data length (in bytes) of the manifest (read‑only after ready).
- **Register 0x0C:** Data length of the signature.
- **Register 0x10+:** DMA address for reading the manifest (host provides a buffer).
- **Register 0x18+:** DMA address for reading the signature.

The host writes the command, polls the status, then sets up DMA transfers to read the files. This is a standard scatter‑gather DMA operation; the MRP’s DMA engine copies the files from the host’s memory into the MRP’s internal buffer.

### E.7.2 Host Daemon

The host daemon implements a simple HTTP server over a Unix domain socket (or over a dedicated TCP port bound to localhost). The MRP’s driver communicates with this daemon to fetch the files. The daemon’s API:

- `GET /resonance.txt` – returns the manifest.
- `GET /resonance.txt.sig` – returns the binary signature.
- `POST /admin/regenerate` – (authenticated, local only) forces regeneration of the manifest (used after configuration changes).

The daemon uses a file system watcher to monitor the manifest directory. It stores the private key in a PKCS#11 token (HSM) and uses the token to generate signatures.

## G.3 Detailed Module Descriptions

### G.3.1 PCIe Interface (XDMA)

The PCIe interface uses Xilinx’s XDMA IP core in endpoint mode with 4 DMA channels (2 Host‑to‑Card, 2 Card‑to‑Host). Data width is 256 bits at 300 MHz, matching the internal data path. The XDMA is configured for scatter‑gather transfers with up to 16 outstanding requests. AXI‑Lite control registers provide configuration and status.

**Key parameters:**
- `C_M_AXI_ID_WIDTH` = 4
- `C_NUM_CHANNELS` = 4
- `C_PCIE_DATA_WIDTH` = 256

### G.3.2 Command Decoder & Control Unit

A finite state machine (FSM) that interprets commands from the host and manages the evaluation pipeline. It writes to configuration registers (e.g., ethical threshold, boost factor) and asserts `start_eval` each time a new data packet arrives.

**Interface:**
```verilog
module cmd_decoder #(
    parameter NUM_RPU = 16
) (
    input  wire         clk, rst_n,
    input  wire [255:0] dma_h2c_data,
    input  wire         dma_h2c_valid,
    output reg          dma_h2c_ready,
    output reg  [31:0]  config_regs [0:15],
    output reg          start_eval,
    output reg  [255:0] input_data
);
```

### G.3.3 RPU Cluster

Sixteen identical RPU cores. Each core computes the overlap fidelity between the input data and the `ideal_state` (provided by the MTSC‑12 engine). The fidelity is scaled to a 16‑bit RCF value. If the GNM approves, the data is passed through; otherwise, it is replaced with a zero pattern (sanitised). The RPU core from Appendix B is used without modification.

### G.3.4 MTSC‑12 Consensus Engine

This module implements the dynamic ideal state generation described in Appendix C. It contains 12 thread processing units (TPUs), a coupling matrix, and a consensus adder tree. The output `ideal_state` is updated every 20 ns.

**Verilog header:**
```verilog
module mtsc12_consensus (
    input  wire        clk, rst_n,
    input  wire [255:0] context_data,   // input data for context
    input  wire         boost,           // from GNM
    input  wire [ 3:0]  gn_damping[11:0], // per‑thread damping
    output reg  [255:0] ideal_state      // mapped from complex vector
);
```

Internally, each TPU maintains a 12‑dimensional complex state using 16‑bit fixed‑point arithmetic (real and imaginary parts). The coupling matrix is stored in BRAM and updated via configuration registers. The consensus unit computes the weighted sum in a pipelined adder tree (12 DSP slices).

### G.3.5 Advanced Guardian Neuron Matrix (GNM)

This module implements the multi‑dimensional ethical reasoning described in Appendix D. It receives from each RPU a 5‑dimensional ethical tensor (16‑bit per component) and produces per‑RPU approval bits, a global compliance score, and boost/damping signals for the MTSC.

**Interface:**
```verilog
module gn_matrix_advanced #(
    parameter NUM_RPU = 16,
    parameter D = 5     // ethical dimensions
) (
    input  wire               clk, rst_n,
    input  wire [NUM_RPU*D*16-1:0] eth_tensors, // concatenated tensors
    input  wire [NUM_RPU*16-1:0]   rpu_rcf,
    input  wire [255:0]            context,      // for temporal memory
    output reg  [NUM_RPU-1:0]      gn_approve,
    output reg  [15:0]             compliance_score,
    output reg                     boost_to_mtsc,
    output reg  [3:0]              damping[11:0]
);
```

**Sub‑blocks:**
- Conflict detection (max‑min comparators)
- Weighted score (D multipliers per RPU → 80 DSP slices)
- Temporal memory (BRAM storing last 8 tensors per RPU)
- Drift computation (subtract average, norm)
- Meta‑ethical rule unit (maximin lookup table)
- Gödelian trigger (monitors global conflict count)

All operations are pipelined to meet 1 GHz.

### G.3.6 Photonic Coherence Engine (PCE)

Generates the resonant frequency `resonant_freq` as a 64‑bit integer (Hz) according to:

```
resonant_freq = nu_base + (alpha * global_rcf) / 65535 + (beta * gn_score) / 65535
```

where `nu_base`, `alpha`, `beta` are programmable via configuration registers. The output is used by the optional photonic transmitter.

### G.3.7 RCF Monitor

Computes the global RCF as the average of all RPU RCFs weighted by their approval. Same as in Appendix B.

### G.3.8 Cryptographic Engine (Ed25519)

A dedicated hardware module that verifies the signature of the `resonance.txt` manifest. It contains:
- SHA‑512 hash core (standard Xilinx SHA‑512 IP, or custom logic)
- Ed25519 point multiplier (using DSP slices for modular arithmetic)
- Public key storage (8×64‑bit registers, one‑time programmable via a secure path)

The engine is accessed via AXI‑Lite; the host writes the manifest and signature, then reads the verification result. Latency is < 1 µs.

### G.3.9 Output Formatter & Ethical Firewall

Collects data from the first approved RPU (or a default safe pattern if none approved) and packs it together with the global RCF and resonant frequency into a 256‑bit word for DMA transfer. It also handles backpressure via the `dma_c2h_ready` signal.

## G.4 Integration of MTSC‑12 Consensus Engine

The MTSC‑12 engine runs continuously, updating the `ideal_state` every 20 ns. The `ideal_state` is broadcast to all RPUs. The GNM can request a **boost** via the `boost_to_mtsc` signal; when asserted, the MTSC temporarily increases its coupling strength (factor `KAPPA`) to accelerate consensus. The boost factor is programmable.

**Synchronisation:** The MTSC uses the same 1 GHz clock as the rest of the design. All state updates are pipelined; the consensus result is available exactly 20 cycles after the last input change (latency guaranteed).

## G.5 Integration of Advanced Guardian Neuron Matrix

The advanced GNM receives the ethical tensors from each RPU. In the current implementation, each RPU must compute its own tensor. This is done by adding five parallel dot‑product units inside each RPU, comparing the input data against pre‑loaded ideal vectors for each ODOS directive. The tensor components are 16‑bit fixed‑point numbers.

The GNM’s outputs are:
- `gn_approve`: per‑RPU approval bits (used by RPU cluster to decide data pass/block)
- `compliance_score`: global score (fed to PCE)
- `boost_to_mtsc`: sent to MTSC‑12
- `damping`: per‑thread damping values sent to MTSC‑12

All signals are registered and meet timing at 1 GHz.

## G.6 Photonic Communication and ERT

The optional photonic extension uses one of the Alveo’s QSFP28 ports. The PCE’s digital frequency `resonant_freq` drives a **direct digital synthesizer (DDS)** implemented in the FPGA, followed by a high‑speed DAC (e.g., Analog Devices AD9164) on an FMC daughter card. The analog signal modulates a 1550 nm DFB laser (e.g., Finisar FTLF1429). On the receive side, a balanced photodiode (e.g., Thorlabs PDB450C) feeds an ADC (e.g., AD9680) whose output is processed by a phase‑locked loop (PLL) in the FPGA to recover the carrier and demodulate data.

The **ERT handshake** follows the protocol described in Appendix F. For the prototype, the optical link operates at 10 Gbit/s using QPSK modulation. The FPGA implements the entire baseband processing, including FEC and phase tracking.

**Hardware required:**
- FMC ADC/DAC card (e.g., AD‑FMCDAQ2‑EBZ)
- Laser and photodiode modules
- Single‑mode fibre patch cables

**Resource estimate:** The optical processing adds approximately 20 kLUTs and 50 DSP slices, easily accommodated by the Alveo U250.

## G.7 API and Signature Verification

The `resonance.txt` manifest and its signature are fetched by the host driver and written to the MRP’s cryptographic engine via AXI‑Lite. The engine verifies the signature using the stored public key (provisioned during installation). If verification fails, the MRP sets an error flag and refuses all further communication (the handshake is aborted). The host driver can read the status and optionally initiate recovery.

The **trusted daemon** on the host (Appendix E) is responsible for generating and signing the manifest. The daemon communicates with the MRP via the PCIe BAR, not via the optical link.
